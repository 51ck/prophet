# Roadmap

Product milestones only. Engineering detail lives in [tech/](../tech/AGENTS.md) after this idea pack.

## Phase 0 — Spec

Pure-idea DOX pack in `spec/`: vision, session, deck/ritual, decks, character, memory, agent, spreads, Telegram UX, economics, this roadmap.

**Done when:** idea docs describe the product without implementation detail; open UX/econ questions stay labeled open.

## Phase 1 — Ritual core

- Light Seer’s full deck (prophet’s preferred)
- Starter spreads
- Character + memory behavior (recall → find question → offer deck → ritual → close → refactor)
- DM-only Telegram path with Phase 1 UX locks
- Honest shuffle / draw / open; mid-ritual Q&A without breaking card state
- Telegram product gap after core MVP (buttons, formatting, soft profile) — board: [tech/telegram-tasks.md](../tech/telegram-tasks.md) per [tech/tickets.md](../tech/tickets.md)

**Done when:** a seeker can complete a real reading whose cards come only from ritual, with continuity of memory and a person-like prophet.

## Phase 2 — Prophet voice

- Remaining decks fully described
- Richer offer logic (question + behavior + fluent memory of past deck)
- Stronger mid-ritual dialogue and deck-appropriate register

**Done when:** deck offers feel intentional and readings differ honestly by deck, not by invented outcomes.

## Phase 3 — Telegram fluency

- DM polish
- Group summon rules and privacy model
- Ceremonial reveal UX beyond Phase 1 text cues

**Done when:** DM readings feel fluid; group summon has explicit documented etiquette.

## Phase 4 — Stewardship (before monetization)

Public presence, operator visibility, and cost control. Prepare for paid access without changing authenticity rules. All of the below land **before** Phase 5.

- **Manifest** — product/public statement seekers and stores can point at. Content composed later; placeholder in roadmap until written.
- **Website** — public home for description, terms and conditions, and related legal/product copy (not the reading UX).
- **Statistics** — usage and cost visibility so stewardship decisions rest on numbers, not guesswork.
- **Admin panel** — operator surface alongside the core: track usage, spot abuse/cost spikes, support limits. Not a seeker-facing channel.
- Usage limits, abuse and cost control (product answers for runaway free use).

**Done when:** manifest + website exist with terms/description; operators can see usage stats via admin; runaway cost/abuse paths have product answers; free unlimited drift is not the default forever.

## Phase 5 — Economics

Monetization experiments — likely Telegram Stars or equivalent — per [economics.md](economics.md). Pay for access/capacity, never for nicer cards. Requires Phase 4 public/legal and operator visibility in place.

**Done when:** a sustainable access model exists that seekers understand and that leaves ritual honesty intact.

## Ordering rule

Do not pull Phase 5 ahead of proven ritual + usable Telegram feel + Phase 4 stewardship (manifest, website/terms, stats, admin). Stewardship may overlap late Phase 3, but authenticity never waits on payment. Character and memory are in from Phase 1 — not an afterthought.
