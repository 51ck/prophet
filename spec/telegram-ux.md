# Telegram UX

## Surfaces

- **Direct message (primary)** — intimate reading space; full session arc lives here
- **Group summon** — prophet can be present in a chat; summon starts a path into a reading. Exact etiquette (in-group vs pull to DM, who may see cards) is open — see below

## Fluency goals

- Natural language first; few mandatory commands
- Ritual progress visible without message spam
- Card reveals feel ceremonial (pace and framing matter more than chrome)
- Seeker always knows: are we in intake, ritual, or closing?
- Mid-ritual questions feel natural in the same thread as the reading

## Interaction principles

- Prefer short prophet turns over walls of text
- One clear next step when the seeker might be stuck (lock question, cut, open next)
- Optional controls (buttons or quick replies) only when they speed a ritual choice — not as a menu maze
- Group: protect seeker privacy by default until a deliberate design says otherwise

## Open questions

Not decided yet — do not pretend solved:

1. **Group reading locus** — full ritual in group, or summon → continue in DM?
2. **Reveal chrome** — text-only names, images, progressive edits of one message, or a short thread of opens?
3. **Controls** — pure chat vs light buttons for cut / open next / lock question?
4. **Interruptions** — how group noise or multiple speakers affect a single reading
5. **Session resume** — if the seeker leaves mid-ritual, resume vs fresh start?
6. **Visibility of deck state** — how much “inspectability” to show seekers without breaking ceremony

## Non-goals (for now)

- Mini-apps or external web UI as the main interface
- Dense command menus as the default path
- Public performance readings without an explicit privacy model
